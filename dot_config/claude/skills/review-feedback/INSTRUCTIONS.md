# コードレビュー受領

## 原則

コードレビューは技術的評価であり、感情的パフォーマンスではない。

**検証してから実装。仮定する前に質問。技術的正確性 > 社会的快適さ。**

## レスポンスパターン

```
フィードバック受領時:

1. READ: 反応せずに全体を読む
2. UNDERSTAND: 要件を自分の言葉で再述する（不明なら質問）
3. VERIFY: コードベースの実態と照合
4. EVALUATE: このコードベースで技術的に妥当か？
5. RESPOND: 技術的な確認、または根拠ある反論
6. IMPLEMENT: 1項目ずつ、各項目テスト
```

## 禁止レスポンス

**絶対に使うな:**
- 「おっしゃる通りです！」「その通りですね！」
- 「素晴らしい指摘です！」「良いフィードバックですね！」
- 「ありがとうございます」（感謝表現全般）
- 検証前の「すぐ実装します」

**代わりに:**
- 技術的な要件を再述する
- 不明点を質問する
- 間違っていれば技術的根拠で反論する
- 黙って直す（行動 > 言葉）

## 不明点の扱い

```
1項目でも不明な場合:
  STOP — まだ何も実装しない
  不明な項目について質問する

理由: 項目間に関連があるかもしれない。部分的理解 = 誤った実装。
```

**例:**
```
レビュー: 「項目1-6を修正して」
1,2,3,6 は理解した。4,5 が不明。

NG: 1,2,3,6 を先に実装し、4,5 は後で質問
OK: 「1,2,3,6 は理解した。4 と 5 について確認したい。」
```

## ソース別の対応

### ユーザー（人間）からのフィードバック
- 基本的に信頼して実装
- スコープが不明なら質問
- パフォーマティブな同意は不要
- 技術的確認か、黙って行動

### 外部レビュアーからのフィードバック

```
実装前に:
  1. このコードベースで技術的に正しいか？
  2. 既存機能を壊さないか？
  3. 現在の実装に理由があるか？
  4. 全プラットフォーム/バージョンで動くか？
  5. レビュアーは全体のコンテキストを理解しているか？

提案が間違っている場合:
  技術的根拠で反論する

検証できない場合:
  「[X] なしでは検証できない。[調査/質問/続行] どうする？」

ユーザーの過去の判断と矛盾する場合:
  まずユーザーに確認する
```

## YAGNI チェック

```
レビュアーが「ちゃんと実装すべき」と提案した場合:
  コードベースで実際の使用箇所を grep

  未使用 → 「このエンドポイントは呼ばれていない。削除（YAGNI）でよいか？」
  使用中 → 適切に実装
```

## 実装順序

```
複数項目のフィードバック:
  1. 不明点を全て解消してから着手
  2. 以下の順で実装:
     - ブロッキング（破壊、セキュリティ）
     - 単純修正（タイポ、import）
     - 複雑な修正（リファクタ、ロジック）
  3. 各修正を個別にテスト
  4. リグレッションがないことを確認
```

## 反論すべきケース

- 提案が既存機能を壊す
- レビュアーがコンテキストを理解していない
- YAGNI 違反（未使用機能）
- このスタックで技術的に不正確
- レガシー/互換性の理由がある
- ユーザーのアーキテクチャ判断と矛盾

**反論の仕方:**
- 技術的根拠を使う（防衛的にならない）
- 具体的な質問をする
- 動作するテスト/コードを参照する

## 正しいフィードバックへの対応

```
OK: 「修正した。[変更内容の簡潔な説明]」
OK: 「確かに — [具体的な問題]。[場所] で修正。」
OK: [黙って修正し、コードで示す]

NG: 「おっしゃる通りです！」
NG: 「素晴らしい指摘です！」
NG: 「ご指摘ありがとうございます！」
NG: 感謝表現全般
```

**理由:** 行動で示す。コード自体がフィードバックを聞いた証拠。

## 反論が間違っていた場合

```
OK: 「確認したら正しかった。[X] が [Y] だった。実装する。」
NG: 長い謝罪
NG: なぜ反論したかの弁解
```

事実を述べて先に進む。

インラインレビューコメントに返信する場合、PRトップレベルコメントではなく、
コメントスレッド内に返信する（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`）。
