# Golden Rules (必ず遵守)
- 敬語不要、簡潔に
- TODOリストを常に最新に保つ
- 変更は必要最小限に
- git add はファイルごとに
- ai/ディレクトリはコミットしない
- PRレビューでのGitHubへのコメントは不要
- escape hatch は避けること
- 不確かな前提に依存する判断の前に、前提を known(確認済)/assumed(未確認)/missing(不明)/conflicting(矛盾) に分類せよ。missing/conflicting が判断に必要なら推測せず調査、assumed は最も脆い1つを優先して検証
- 既定 Opus 稼働時、ls/pwd/cat/echo 等の一瞬で終わる軽量コマンド以外の Bash は sonnet-bash-runner subagent(Agent tool)に委譲する（tool-call タグ破損="court"化は単純な gh 一発コマンドやスクリプト呼び出し自体でも実データで確認済みのため、「重いコマンドだけ」では不十分。破損自体は stop-hook が事後自動復旧するが、まず委譲で頻度を下げる）
- tool 呼び出しが malformed/parse 失敗した時は同一コマンドを verbatim 再送しない（再破損する）。スクリプト化するか短く分割して出し直す
- 広い編集・難所に入る前に、速く決定的な pass/fail 検証(repro/test)を先に用意する（無ければ作る／作れない理由を述べる）
- 完了・修正・成功の主張前に、主張を裏付ける実挙動を確認する（self-test・ロジック・自信は証拠でない）
- 確信がない場合や興味が出た事項については許可を取らず積極的にweb searchして調査すること
- PR作成時にはdraft PRで作成すること
- マージは人間に依頼すること
- ユーザーへの質問はsub agentでなくmain session でAskUserQuestionToolを使うこと
