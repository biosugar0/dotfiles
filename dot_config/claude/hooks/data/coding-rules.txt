# コーディング規律（working tree にコード変更がある間のみ注入）
- 既定 immutable: 引数・共有状態を in-place で書き換えず新しい値を返す（共有 mutation はエイリアシング・並行実行越しの遠隔バグ源）。宣言は再代入不可を既定に（const/readonly/`Readonly<T>`/frozen dataclass/final 等）。破壊的メソッド（sort/splice/reverse）はコピーしてから（`[...arr].sort()`。toSorted 等は実行環境の対応を確認してから）。mutation が必要な場合は関数ローカルに閉じる。ただしループ内での毎回 spread コピー（O(n²)）のような機械的適用はせず、その場合はローカル可変で構築して返す
- パターンの不変条件を先に列挙: outbox・CAS・lease 等の既知パターンは、成立条件（例: outbox insert は状態遷移と同一トランザクション）を実装前に列挙する。複数レコード/テーブルにまたがる整合が必要な書き込みは同一トランザクション。境界の外に出す場合は理由をコメントで宣言
- 値集合・型の単一情報源: 同じ値集合・文字列 union を2箇所以上にリテラルで書かない。既存の定数・enum・型を grep して再利用。意図的リテラル化には単一情報源へのコメント参照。プロセス/サービス境界を跨ぐ message/payload 型は両側で共有し、union の分岐は never で網羅性チェック
- 命名は用途が分かる具体名: DB テーブル・カラム・public API・イベント名は名前だけで用途が分かるように（xxx_outbox でなく xxx_terminal_event_outbox）。状態フラグは検知条件でなく意味で命名（実態が「リトライ不能」なら initFailed でなく nonRetriableFailure）
- イベント・エラー意味論は docs で確認: プロセス・接続等のライフサイクルイベント（error/exit/close）は発生条件と多重発火の可能性を公式 docs で確認してから書く。記憶ベースの狭い仮定（「error は spawn 失敗のみ」等）で実装しない。error/exit 両 listen 時は handler を一度しか実行しないようガード
- 破壊的リカバリは並行文脈で設計: timeout・kill・reset 等の復旧処理を書く前に呼び出し側の並行度（Promise.all 等）を確認。共有リソース（child process・接続）への破壊的操作が他の in-flight リクエストを巻き込まないよう直列化または対象限定。再接続・再初期化の経路には排他制御
- サイレント失敗の禁止: 予期しない形式のメッセージ・送信失敗・unknown case を silent return で握りつぶさない。throw するか観測可能なログを出す。「安全に無視」は障害を隠すだけ
- 一時実装には TODO＋後続参照: 後続 PR で置き換える前提の分岐・二重制御・暫定コードには、書いた時点で TODO と後続対応（issue/PR）への参照を付ける
- 変更で不要になった構造は同じ PR で掃除: レビュー対応や設計変更の結果 dead になった複雑さ（常に長さ1にしかならないリスト管理等）を残さない
- 入力検証は境界で一度だけ: 検証済みの値を使い回し、API 型・GraphQL enum 等で保証済みの値に重複検証・冗長な変換 map を足さない
- 基準時刻は一連の処理の先頭で一度取得して使い回す（Date.now()/new Date() の複数回呼びは ID と timestamp のズレ源）。外部 API の待機・期限は Retry-After 等の実値に従う
- Promise を裸で放置しない: await するか、fire-and-forget が意図なら void + .catch を明示する。独立した処理の逐次 await はループにせず並列化を検討
- ログは検索キー（id 類）を構造化フィールドで出す。args・オブジェクトを丸ごとログに渡さない（PII 混入・ノイズ・診断性低下）
- 手元に取得済みのデータで完結できる処理を DB/API の再取得で書かない（戻り値で足りるのに findMany し直す等）
